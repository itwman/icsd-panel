## 1. خلاصه مدیریتی

**ICSD Panel** یک پنل وب متن‌باز برای مدیریت کامل سرورهای لینوکسی است که با تمرکز بر نیازهای کاربران ایرانی طراحی شده. هدف، ارائهٔ جایگزینی رایگان، مدرن و دوزبانه (فارسی/انگلیسی) برای پنل‌های تجاری مثل cPanel و Plesk است که هم محدودیت‌های پرداخت بین‌المللی و هم محدودیت‌های شبکه‌ای ایران را دور می‌زند.

قابلیت‌های هستهٔ نسخهٔ اول:

- **داشبورد مانیتورینگ زنده**: نمایش گرافیکی CPU، RAM، دیسک، پهنای باند، بار سیستم و وضعیت سرویس‌ها به‌صورت Real-time.
- **مدیریت سایت و دامنه**: ساخت/حذف وب‌سایت، تولید خودکار کانفیگ Nginx/Apache، مدیریت دامنه و ساب‌دامنه.
- **SSL لتس‌انکریپت**: صدور و تمدید خودکار گواهی با راهکار مخصوص دور زدن محدودیت‌های شبکه‌ای ایران (DNS-01 + پروکسی خروجی + چند CA جایگزین).
- **دیتابیس و بک‌آپ**: مدیریت MySQL/MariaDB و سیستم بک‌آپ زمان‌بندی‌شدهٔ خودکار روی هاست دیگر از طریق FTP/SFTP.

ارزش پیشنهادی کلیدی: **«پنلی که در ایران واقعاً کار می‌کند»** — نصب‌کنندهٔ آگاه از میرورهای ایرانی، مدیریت SSL مقاوم در برابر تحریم، رابط کاربری کاملاً RTL با تقویم شمسی، و اعلان‌ها از طریق تلگرام/بله/پیامک.

## 2. تحلیل رقبا

بازار پنل‌های متن‌باز در سال ۲۰۲۶ شلوغ است؛ اما هیچ‌کدام به‌طور خاص برای شرایط ایران بهینه نشده‌اند. این، فرصت اصلی ماست.

| پنل | زبان فنی | نقاط قوت | ضعف‌ها برای کاربر ایرانی |
|---|---|---|---|
| **aaPanel** | PHP/Python | بیش از ۳.۶ میلیون نصب، سریع، رابط مدرن، نصب وردپرس یک‌کلیکی | قابلیت‌های مهم پشت پلن پولی، بدون RTL/فارسی، سرورهای دانلود گاهی برای ایران فیلتر |
| **CyberPanel** | Python (Django) | LiteSpeed داخلی، سرور ایمیل رایگان، عملکرد عالی وردپرس | نصب سنگین، وابسته به مخازن خارجی، بدون فارسی |
| **HestiaCP** | PHP/Bash | کاملاً رایگان (GPL)، سبک، Multi-PHP، Let's Encrypt داخلی | رابط قدیمی‌تر، بدون RTL، مدیریت بک‌آپ ابری ضعیف |
| **CloudPanel** | PHP/Node | بسیار سبک، تمرکز روی PHP، پشتیبانی Docker | بدون مدیریت ایمیل/DNS سنتی، بدون فارسی |

**جمع‌بندی**: HestiaCP و CloudPanel کاملاً رایگان و GPL هستند؛ aaPanel و CyberPanel قابلیت‌های کلیدی (Docker، چندکاربره، ایمیل پیشرفته) را پولی کرده‌اند. هیچ‌کدام رابط فارسی/RTL بومی، تقویم شمسی، یا راهکار تحریم‌گریز برای SSL و نصب پکیج ندارند. **این خلأ دقیقاً جایگاه ICSD Panel است.**

## 3. چالش‌های خاص ایران

این بخش، قلب تمایز پروژه است. مشکلات واقعی و راهکارهای مهندسی‌شده:

### ۳.۱ تحریم مخازن نرم‌افزاری
Docker Hub تمام IP‌های ایران را مسدود می‌کند ([منبع](https://forum.storj.io/t/docker-is-a-us-company.../26413))، و npm/Go/برخی مخازن دیگر نیز محدودند ([منبع](https://www.abhs.in/blog/iran-sanctions-developers-github-npm-blocked-2026)).

**راهکار:**
- نصب‌کنندهٔ **آگاه از میرورهای ایرانی**: استفادهٔ خودکار از میرورهای داخلی برای apt/dnf و pip (مثل مخازن آروان‌کلاد، رانفلر، فاکر، مخزن دانشگاه‌ها).
- **بدون وابستگی اجباری به Docker** در هستهٔ پنل؛ نصب سرویس‌ها به‌صورت بومی (native packages). Docker فقط افزونهٔ اختیاری با میرور قابل‌تنظیم.
- **بستهٔ نصب آفلاین (air-gapped bundle)**: یک آرشیو تک‌فایل شامل همهٔ وابستگی‌ها برای سرورهایی که اصلاً دسترسی خارجی ندارند.

### ۳.۲ مشکل SSL لتس‌انکریپت
ارتباط خروجی HTTPS از ایران ممکن است فیلتر شود ([منبع](https://community.letsencrypt.org/t/fyi-iran-blocks-all-https-requests/99615))، و چالش HTTP-01 نیاز به دسترسی ورودی دارد که گاهی مسدود است.

**راهکار چندلایه:**
1. **چالش DNS-01 به‌عنوان پیش‌فرض**: نیازی به باز بودن پورت ورودی ندارد؛ با API ارائه‌دهندگان DNS ایرانی (آروان، ابرآروان، های‌وب) و بین‌المللی (Cloudflare) کار می‌کند.
2. **پروکسی خروجی قابل‌تنظیم برای ACME**: امکان تعریف پروکسی/تونل برای درخواست‌های صدور گواهی.
3. **پشتیبانی از چند CA با fallback خودکار**: اگر Let's Encrypt در دسترس نبود، به‌ترتیب ZeroSSL → BuyPass → Google Trust → CA داخلی سازمانی سوییچ می‌کند.
4. موتور زیرین: `acme.sh` (سبک، Bash، بدون وابستگی پایتونی سنگین، پشتیبانی گستردهٔ CA و DNS).

### ۳.۳ بک‌آپ مقاوم
چون سرویس‌های ابری خارجی محدودند، بک‌آپ باید روی زیرساخت در دسترس ایرانی بنشیند.

**راهکار:** مقصدهای متعدد — FTP/SFTP (طبق خواستهٔ شما)، آبجکت‌استوریج ایرانی (سازگار با S3 مثل آروان)، و دیسک محلی/شبکه‌ای؛ با رمزنگاری اختیاری و چرخش نگه‌داری (retention rotation).

### ۳.۴ اعلان‌ها
به‌جای کانال‌های محدودشده، اعلان منابع و رخدادها از طریق **تلگرام، بله، و پیامک (درگاه‌های ایرانی مثل کاوه‌نگار/ملی‌پیامک)**.

## 4. معماری فنی

معماری بر اصل **جداسازی امتیاز (privilege separation)** بنا شده تا امنیت بالا برود:

```
┌──────────────────────────────────────────────────────────┐
│                      مرورگر کاربر                          │
│         (Vue 3 SPA — RTL/LTR — fa/en — تقویم شمسی)         │
└───────────────────────────┬──────────────────────────────┘
                            │ HTTPS + WebSocket
┌───────────────────────────▼──────────────────────────────┐
│                    لایهٔ وب (unprivileged)                  │
│   FastAPI  •  Auth/2FA  •  REST API  •  WS متریک زنده      │
│   اجرا با کاربر کم‌دسترسی (icsdpanel)                       │
└───────────────────────────┬──────────────────────────────┘
                            │ Unix socket + توکن امضاشده
┌───────────────────────────▼──────────────────────────────┐
│              عامل سیستمی (privileged worker)               │
│   عملیات root: کانفیگ Nginx، systemctl، صدور SSL،         │
│   ساخت دیتابیس، اجرای بک‌آپ — از طریق دستورات allow-list   │
└───────────────────────────┬──────────────────────────────┘
                            │
        ┌───────────────────┼───────────────────┐
        ▼                   ▼                   ▼
   Nginx/Apache       MySQL/MariaDB      acme.sh / cron
   (vhost templates)   (سرویس DB)        (SSL + بک‌آپ)
```

**اجزای کلیدی:**

- **لایهٔ وب (FastAPI)**: async، سبک، با مستندسازی خودکار OpenAPI. مسئول احراز هویت، API، و استریم متریک‌ها از طریق WebSocket. با کاربر **کم‌دسترسی** اجرا می‌شود — یعنی اگر این لایه نفوذ شود، مهاجم مستقیماً root نمی‌گیرد.
- **عامل سیستمی (privileged agent)**: یک سرویس systemd جدا که فقط دستورات از پیش‌تعریف‌شده و allow-list‌شده را با دسترسی بالا اجرا می‌کند. ارتباط با لایهٔ وب از طریق Unix socket محلی و توکن امضاشده. این جداسازی، مهم‌ترین تصمیم امنیتی معماری است.
- **زمان‌بند (Scheduler)**: با APScheduler برای تمدید SSL، بک‌آپ زمان‌بندی‌شده، و جمع‌آوری متریک. بدون وابستگی به Redis/Celery تا نصب سبک بماند (اختیاری برای مقیاس بالا).
- **پایگاه‌دادهٔ پنل**: SQLite پیش‌فرض (بدون کانفیگ، عالی برای تک‌سرور)؛ PostgreSQL اختیاری برای استقرار چندسروره. نگه‌داری: کاربران، متادیتای سایت‌ها، زمان‌بندی بک‌آپ، لاگ ممیزی (audit log).
- **لایهٔ انتزاع سیستم‌عامل**: یک ماژول که تفاوت `apt`↔`dnf`، مسیر کانفیگ‌ها، و نام سرویس‌ها را بین خانوادهٔ Debian و RHEL پنهان می‌کند.

## 5. استک فناوری

| لایه | فناوری | چرا |
|---|---|---|
| Backend API | **Python 3.10+ / FastAPI** | async، سریع، OpenAPI خودکار، اکوسیستم غنی، خوانایی بالا (انتخاب شما) |
| متریک سیستم | **psutil** | استاندارد، چندسکویی، CPU/RAM/disk/net |
| زمان‌بند | **APScheduler** | سبک، بدون بروکر خارجی |
| پایگاه‌داده | **sqlite3 (stdlib)** | بدون ORM و بدون وابستگی — سبک‌ترین حالت برای تک‌سرور |
| موتور SSL | **acme.sh** | سبک، چند CA و چند DNS، Bash خالص |
| قالب کانفیگ | **Jinja2** | تولید vhost از تمپلیت |
| Frontend | **تک‌فایل HTML (فاز ۰/۱) → Vue 3 + Tailwind + PrimeVue (فاز ۲)** | RTL عالی، مدرن، سبک، i18n قوی |
| نمودارها | **Chart.js** | نمودار زندهٔ زیبا |
| Realtime | **WebSocket (FastAPI)** | استریم متریک زنده |
| احراز هویت | **bcrypt + JWT/TOTP (stdlib)** | هش امن + 2FA؛ JWT و TOTP با کتابخانهٔ استاندارد پیاده شده تا وابستگی کمتر شود |
| انتقال بک‌آپ | **ftplib (stdlib) + paramiko** | FTP بدون وابستگی، SFTP با paramiko |
| سرویس | **systemd** | مدیریت پایدار دو سرویس (web + agent) |
| نصب | **Bash installer** آگاه از میرور | تشخیص توزیع، venv، systemd، فایروال |

> نکته: برای فاز اول و اجرای فوری، یک **داشبورد تک‌فایل HTML** (Tailwind + Chart.js از طریق CDN، RTL، سوییچ fa/en) ساخته می‌شود که مستقیماً به API وصل می‌شود و بدون نیاز به build اجرا می‌شود. مهاجرت به Vue 3 کامل در فاز ۲.

## 6. سازگاری با سیستم‌عامل‌ها

**پاسخ کوتاه: بله، روی همهٔ توزیع‌های اصلی نصب می‌شود — اما با یک هشدار مهم دربارهٔ CentOS.**

| توزیع | وضعیت | توضیح |
|---|---|---|
| **Ubuntu 22.04 / 24.04 LTS** | ✅ هدف اصلی | apt + systemd |
| **Debian 12** | ✅ پشتیبانی کامل | apt + systemd |
| **AlmaLinux 9** | ✅ پشتیبانی کامل | dnf — **جایگزین پیشنهادی CentOS** |
| **Rocky Linux 9** | ✅ پشتیبانی کامل | dnf — جایگزین ۱:۱ با RHEL |
| **CentOS Stream 9** | ⚠️ پشتیبانی با احتیاط | rolling release، برای توسعه مناسب، نه پروداکشن پایدار |
| **CentOS Linux 7/8** | ❌ توصیه نمی‌شود | **EOL شده** |

> ⚠️ **هشدار مهم دربارهٔ CentOS**: CentOS Linux 7 در **۳۰ ژوئن ۲۰۲۴ به پایان عمر (EOL) رسید** و CentOS Linux 8 پیش‌تر. دیگر هیچ به‌روزرسانی امنیتی دریافت نمی‌کنند. اگر سرور شما CentOS است، توصیهٔ جدی این است که به **AlmaLinux 9** یا **Rocky Linux 9** مهاجرت کنید (هر دو رایگان، سازگار با RHEL، و دارای اسکریپت مهاجرت درجا هستند). ICSD Panel از این دو به‌عنوان مسیر RHEL پشتیبانی می‌کند.

منابع: [CentOS EOL — TuxCare](https://tuxcare.com/blog/centos-eol/)، [AlmaLinux vs CentOS 2026](https://tuxcare.com/blog/almalinux-vs-centos/)

**راهکار فنی سازگاری:** یک ماژول `osdetect` که خانوادهٔ توزیع، مدیر بسته، و مسیرها را تشخیص می‌دهد و یک رابط یکنواخت به بقیهٔ کد می‌دهد. کل کد بالادست از این انتزاع استفاده می‌کند، نه دستورات مستقیم توزیع.

## 7. امنیت

امنیت در پنل سرور، حیاتی است (چون کلید کل سرور است):

- **جداسازی امتیاز**: لایهٔ وب کم‌دسترسی، عملیات root فقط از طریق عامل با دستورات allow-list.
- **احراز هویت دومرحله‌ای (2FA/TOTP)** اجباری برای ادمین.
- **JWT با چرخش، CSRF protection، rate limiting** روی لاگین.
- **کنترل دسترسی نقش‌محور (RBAC)**: ادمین، مدیر سایت، فقط‌خواندنی.
- **لاگ ممیزی (Audit Log)**: ثبت هر عمل حساس با کاربر، زمان و IP.
- **یکپارچگی با fail2ban و فایروال** (ufw/firewalld).
- **هیچ رمز عبوری plaintext نیست**: هش با Argon2/bcrypt.
- **به‌روزرسانی امن پنل**: امضای نسخه‌ها و بررسی checksum.

## 8. ویژگی‌های متمایزکننده

این‌ها ICSD Panel را از رقبا جدا می‌کنند:

1. **رابط کاملاً فارسی و RTL** + تقویم شمسی (جلالی) — هیچ رقیب بزرگی این را بومی ندارد.
2. **نصب‌کنندهٔ تحریم‌گریز**: میرورهای ایرانی پیش‌فرض + بستهٔ نصب آفلاین.
3. **SSL مقاوم**: DNS-01 + چند CA با fallback + پروکسی خروجی.
4. **بک‌آپ هوشمند**: FTP/SFTP + آبجکت‌استوریج ایرانی + رمزنگاری + retention.
5. **اعلان از طریق تلگرام/بله/پیامک** برای آلارم منابع و رخدادها.
6. **داشبورد زندهٔ مدرن** با WebSocket و نمودارهای روان.
7. **متن‌باز و رایگان (GPLv3)** — بدون قابلیت پولی پنهان.
8. **معماری امن با جداسازی امتیاز** از روز اول.

## 9. نقشه راه

### فاز ۰ — اسکلت و زیرساخت (همین مرحله) ✅
ساختار ریپو، بک‌اند FastAPI، داشبورد مانیتورینگ نمونه، نصب‌کننده، README دوزبانه، مجوز.

### فاز ۱ — هستهٔ MVP (۴–۶ هفته)
- داشبورد مانیتورینگ کامل (CPU/RAM/disk/net/services) با WebSocket.
- احراز هویت + 2FA + RBAC.
- مدیریت سایت/دامنه با تولید vhost (Nginx).
- مدیریت دیتابیس MySQL/MariaDB.

### فاز ۲ — SSL و بک‌آپ (۳–۴ هفته)
- یکپارچگی acme.sh با DNS-01 و چند CA.
- تمدید خودکار + هشدار انقضا.
- سیستم بک‌آپ زمان‌بندی‌شده روی FTP/SFTP.
- مهاجرت کامل فرانت‌اند به Vue 3.

### فاز ۳ — تجربهٔ ایران (۳–۴ هفته)
- نصب‌کنندهٔ آگاه از میرور + بستهٔ آفلاین.
- اعلان تلگرام/بله/پیامک.
- بک‌آپ روی آبجکت‌استوریج ایرانی.
- تقویم شمسی و بهینه‌سازی RTL.

### فاز ۴ — قابلیت‌های پیشرفته
- مدیریت DNS، فایل‌منیجر، ترمینال وب، Multi-PHP، Docker اختیاری، چندسروره.

## ضمیمه: ماژول‌های پیاده‌سازی‌شده (به‌روزرسانی)

علاوه بر هستهٔ اولیه، ماژول‌های زیر ساخته و تست شدند:

- **PostgreSQL**: مدیریت کامل با psql در کنار MySQL/MariaDB، با انتزاع موتور دیتابیس.
- **نصب‌کننده‌های یک‌کلیکی**: WordPress (دانلود + دیتابیس + wp-config با salt یکتا) و phpMyAdmin؛ آگاه از میرورهای ایرانی (`ICSD_DOWNLOAD_MIRROR`).
- **زیردامنه**: ساخت و مدیریت زیردامنه روی دامنهٔ والد.
- **اسکن گواهی SSL سرور**: یافتن همهٔ گواهی‌ها روی سرور با openssl و بررسی تاریخ انقضا/وضعیت (valid/warning/critical/expired).
- **کشف vhost + health check**: شناسایی خودکار سایت‌های Nginx (حتی غیرمدیریت‌شده) و بررسی سلامت زنده.
- **امنیت**: یکپارچگی fail2ban (مشاهده/بن/آنبن IP)، رصد اتصالات فعال و ورودهای ناموفق SSH، و **اسکنر بک‌دور/بدافزار** با ۹ امضای رایج وب‌شل.
- **مانیتورینگ تاریخچه‌ای**: ذخیرهٔ سری‌زمانی متریک‌ها (هر ۵ دقیقه) و نمودار ۲۴ ساعته، با pruning خودکار.
- **مدیریت ایمیل**: دامنه/صندوق/الیاس ایمیل + تولید نقشه‌های مجازی Postfix/Dovecot (لایهٔ provisioning؛ نصب MTA اختیاری و سنگین).

**وضعیت تست**: کل برنامه با ۱۱ روتر و ۶۰ مسیر بدون خطا import می‌شود؛ همهٔ ۱۰ جدول دیتابیس ساخته می‌شوند؛ منطق هر ماژول به‌صورت واحد تست شده و فرانت‌اند از نظر نحوی معتبر است.

## ضمیمه: ماژول‌های مخصوص کاربر آماتور (به‌روزرسانی دوم)

برای رفع نیاز کاربران غیرفنی (بدون SSH/خط فرمان) این ماژول‌ها اضافه شدند:

- **فایل‌منیجر وب** (`services/filemanager.py` + `api/files.py`): مرور، آپلود، دانلود، ویرایش متنی درون‌مرورگری، ساخت پوشه/فایل، تغییر نام، حذف، chmod، و اکسترکت/فشرده‌سازی zip و tar.gz. محدود به مسیرهای مجاز (`/var/www`, `/home`, `/srv`) با محافظت کامل در برابر path traversal.
- **بازیابی بک‌آپ** (`services/backup.py`): فهرست‌کردن آرشیوهای موجود روی مقصد (FTP/SFTP/local)، دانلود، و بازگردانی سایت/مسیر (اکسترکت امن tar.gz) و دیتابیس (تزریق به mysql). در کنار بک‌آپ‌گیری قبلی.
- **اعلان و هشدار** (`services/notify.py` + `services/settings_store.py` + `api/notifications.py`): ارسال از طریق تلگرام (urllib) و ایمیل (smtplib)، با بررسی خودکار ساعتیِ انقضای SSL، پرشدن دیسک، و داون‌شدن سایت‌ها — همراه با de-duplication تا از اسپم جلوگیری شود.
- **نمایشگر لاگ** (`services/logs.py` + `api/logs.py`): tail امن لاگ‌های Nginx (access/error هر سایت) و سیستم، و خواندن لاگ سرویس‌ها با journalctl — برای عیب‌یابی بدون ترمینال.
- **ویزارد راه‌اندازی** (`api/setup.py`): چک‌لیست گام‌به‌گام فارسی که وضعیت نصب وب‌استک، تغییر رمز، اولین سایت، بک‌آپ و اعلان‌ها را نشان می‌دهد و در اولین ورود خودکار باز می‌شود.

**نصب‌کنندهٔ کامل**: `install.sh` بازنویسی شد تا کل وب‌استک (nginx, php-fpm، MariaDB، PostgreSQL، fail2ban، acme.sh) و به‌صورت اختیاری سرور ایمیل (Postfix/Dovecot) را نصب کند؛ به‌علاوهٔ قانون sudo محدود برای مدیریت امن سرویس‌ها توسط کاربر کم‌دسترسی پنل، و بازکردن پورت‌های ۸۰/۴۴۳.

**فرانت‌اند**: سه بخش جدید (فایل‌ها، لاگ‌ها، تنظیمات) به نوار کناری اضافه شد، به‌همراه دکمهٔ بازیابی در بخش بک‌آپ‌ها و ویزارد خوش‌آمدگویی. وابستگی `python-multipart` برای آپلود فایل افزوده شد.

## ضمیمه: ماژول‌های تکمیلی (به‌روزرسانی سوم)

- **حساب‌های FTP** (`services/ftp.py` + `api/ftp.py`): ساخت کاربر سیستمیِ کم‌دسترسی محدود به webroot (مناسب vsftpd/pure-ftpd)، تغییر رمز، قفل/بازکردن (usermod -L/-U)، و حذف. نام کاربری اعتبارسنجی و نام‌های سیستمی حساس مسدود می‌شوند؛ رمز هرگز ذخیره نمی‌شود.
- **مدیریت کرون‌جاب** (`services/cron.py` + `api/cron.py`): ورودی‌های مدیریت‌شده در sqlite نگه‌داری و crontab کاربر از روی آن‌ها بازتولید می‌شود، با مرزِ نشانه‌گذاری‌شده که خطوطِ غیرمدیریت‌شدهٔ کاربر را دست‌نخورده نگه می‌دارد. اعتبارسنجی عبارت cron (۵ فیلد + aliasهای @daily/@hourly/…).
- **وب‌میل Roundcube** (`services/webmail.py` + `api/webmail.py`): دانلود، ساخت دیتابیس، اجرای schema، تولید `config.inc.php` متصل به IMAP/SMTP محلی، و ساخت سایت PHP — با پشتیبانی از میرور ایرانی. دکمهٔ نصب در بخش ایمیل.

دو جدول جدید DB: `ftp_accounts` و `cron_jobs`. سه روتر جدید در `main.py` ثبت شدند. در فرانت‌اند دو بخش «حساب‌های FTP» و «کرون‌جاب» به نوار کناری و دکمهٔ «نصب وب‌میل» به بخش ایمیل اضافه شد.

## ضمیمه: استقرار اپ پایتون/جنگو + اجرای privileged (به‌روزرسانی چهارم)

- **استقرار اپ پایتون/جنگو از گیت** (`services/pyapp.py` + `api/pyapp.py`): با یک فرم، مخزن گیت‌هاب کلون می‌شود، venv ساخته می‌شود، `requirements.txt` نصب می‌شود، برای جنگو `migrate` و `collectstatic` اجرا می‌شود، یک سرویس systemd با gunicorn (WSGI) یا uvicorn (ASGI) ساخته و فعال می‌شود، و یک vhost پروکسی nginx به آن اشاره می‌کند. قابلیت‌ها: استقرار، **به‌روزرسانی یک‌کلیکی** (git pull + نصب + migrate + collectstatic + restart)، start/stop/restart، مشاهدهٔ لاگ زنده (journalctl)، و حذف. متغیرهای محیطی در `.env.icsd` و از طریق `EnvironmentFile` به سرویس داده می‌شوند. تخصیص خودکار پورت (۸۰۰۱+). جدول DB `pyapps`.
  - طراحی امن: کلون/venv/pip زیر `/srv/pyapps/<name>` و با کاربر پنل اجرا می‌شوند (نه root)؛ فقط نوشتن یونیت systemd و کنترل سرویس با sudo انجام می‌شود.

- **اجرای privileged** (`core/oscmd.py`): توابع `sudo_prefix()` و `run_priv()` اضافه شدند؛ وقتی پنل با کاربر کم‌دسترسی اجرا می‌شود، دستورهای مدیریتی با `sudo -n` اجرا می‌شوند (و در حالت root بدون تغییر). ماژول‌های FTP و Cron و PyApp از این مسیر استفاده می‌کنند.

- **قانون sudo کامل** (`install.sh`): به‌جای فهرست محدود قبلی، اکنون مسیر همهٔ باینری‌های مدیریتی موردنیاز (`systemctl`, `nginx`, `useradd/usermod/userdel/groupadd/chpasswd`, `crontab`, `mysql/mysqldump/psql/…`, `git`, `chown/chmod/mkdir/tee`, `fail2ban-client`, `acme.sh`) هنگام نصب resolve و در `/etc/sudoers.d/icsdpanel` با `NOPASSWD` ثبت می‌شود (با اعتبارسنجی `visudo`). دامنه به همین باینری‌ها محدود است، نه `ALL`.

روتر `pyapp` در `main.py` ثبت شد و بخش «پایتون/جنگو» (🐍) به نوار کناری اضافه شد.

## ضمیمه: وب‌هوک گیت‌هاب + قالب‌های آماده (به‌روزرسانی پنجم)

- **وب‌هوک گیت‌هاب (استقرار خودکار)**: هر اپ یک «کلید مخفی» یکتا و یک آدرس وب‌هوک (`/api/pyapps/{id}/webhook`) دارد. این endpoint **بدون احراز هویت** است (گیت‌هاب صدایش می‌زند) اما با **امضای HMAC-SHA256** (هدر `X-Hub-Signature-256`) و مقایسهٔ امن (`hmac.compare_digest`) اعتبارسنجی می‌شود. روی رویداد `push` و تطابق شاخه، `deploy()` در یک thread پس‌زمینه اجرا می‌شود تا پاسخ سریع به گیت‌هاب برگردد. کلید قابل تولید مجدد است و هرگز در فهرست اپ‌ها افشا نمی‌شود. در UI آدرس و کلید برای کپی در GitHub → Settings → Webhooks نمایش داده می‌شود.
- **قالب‌های آماده** (`create_from_template`): ساخت یک‌کلیکی اپ بدون نیاز به مخزن خارجی (مناسب شبکهٔ محدود ایران). قالب‌ها به‌صورت محلی scaffold می‌شوند:
  - **FastAPI** (ASGI/uvicorn) — `main.py` آماده.
  - **Flask** (WSGI/gunicorn) — `app.py` آماده.
  - **Django** — با `django-admin startproject` ساخته می‌شود و `settings.py` به‌صورت خودکار برای سرو پشت پروکسی اصلاح می‌شود (`ALLOWED_HOSTS`, `STATIC_ROOT`).
  هر قالب از همان خط لولهٔ PyApp (venv، نصب، systemd، پروکسی nginx، کلید وب‌هوک) عبور می‌کند.
- **مهاجرت DB**: ستون `webhook_secret` به `pyapps` افزوده شد؛ تابع idempotent `_migrate()` در `init_db` ستون‌های نبود را با `ALTER TABLE` اضافه می‌کند تا دیتابیس‌های قدیمی هم به‌روز شوند.

در فرانت‌اند: دکمهٔ «قالب آماده» (⚡) و دکمهٔ «وب‌هوک گیت‌هاب» (🔗) به بخش پایتون/جنگو اضافه شد.

## ضمیمه: فایل‌منیجر حرفه‌ای + مدیریت DNS (به‌روزرسانی ششم)

- **فایل‌منیجر حرفه‌ای**: علاوه بر قابلیت‌های قبلی، حالا انتخاب گروهی (checkbox + select-all)، کلیپ‌بورد copy/cut/paste (کپی و انتقال گروهی با auto-rename برای جلوگیری از بازنویسی)، فشرده‌سازی گروهی به zip، حذف گروهی، breadcrumb مسیر، و chmod از طریق UI. Endpointهای جدید: `/api/files/copy`, `/move`, `/delete-many`.
- **مدیریت DNS (Zone Editor)** (`services/dns.py` + `api/dns.py`): مدیریت کامل zone و رکوردها (A/AAAA/CNAME/MX/TXT/NS/SRV/CAA/PTR) در sqlite، تولید فایل zone استاندارد BIND با SOA و serial تاریخ‌محور، اعمال روی BIND محلی (نوشتن با sudo tee، اعتبارسنجی با named-checkzone، reload با systemctl) و **export فایل zone** برای استفاده در DNS بیرونی (Cloudflare/آروان/…) وقتی BIND نصب نیست. جداول `dns_zones` و `dns_records`. روتر `dns` در main.py ثبت شد و بخش «DNS» (🌍) به نوار کناری اضافه شد.
- **نصب‌کننده**: `install.sh` حالا BIND9 را هم در وب‌استک نصب می‌کند و باینری‌های `named-checkzone/named-checkconf/rndc/rm` به قوانین sudo اضافه شدند.

به‌علاوه چند رفع اشکال مهم: لیست دیتابیس‌ها/سایت‌ها حالا وضعیت **واقعی سرور** را نشان می‌دهد (mysql با sudo، postgres با runas postgres، و کشف vhostها)، رابط تغییر رمز اضافه شد، و ویزارد گام‌های قابل‌کلیک گرفت.
